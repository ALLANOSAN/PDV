Vamos construir seu sistema de PDV local com foco em funcionalidades essenciais de frente de caixa, integração com pagamentos e gestão de caixa. Este ambiente será construído com React, Vite, TypeScript e Tailwind CSS, utilizando componentes do Shadcn/UI e o backend Supabase para autenticação e, se necessário, funcionalidades como Realtime.

Começaremos pela estrutura base. A aplicação será um Progressive Web App (PWA) rodando localmente, otimizado para performance. O frontend será em React, utilizando Vite para um build rápido e TypeScript para segurança de tipos. Toda a interface visual será estilizada com Tailwind CSS, e para componentes como botões, modais e inputs, utilizaremos o Shadcn/UI, garantindo consistência e acessibilidade.

No coração do sistema, teremos o Supabase como backend. Criaremos as tabelas necessárias no PostgreSQL para armazenar informações de produtos, vendas, transações de caixa (abertura, fechamento, sangrias), e usuários. Implementaremos Row Level Security (RLS) para garantir que cada usuário só acesse seus próprios dados, essencial para a segurança do PDV.

Vamos iniciar com a landing page que servirá como ponto de entrada e apresentação do sistema. Imagine um visual sóbrio e profissional, talvez com um fundo em bg-gray-100 e um esquema de cores focado em tons de azul escuro (text-gray-800) e um verde sutil (#10B981 ou emerald-500) para botões de ação principais, transmitindo confiança e eficiência. O cabeçalho terá o logo do sistema, "PDV Local Pro" em font-bold text-2xl text-gray-800, e um link de "Login" com text-gray-600 hover:text-emerald-500 transition-colors. O hero section focará em um título impactante como "Seu ponto de venda local otimizado" em text-5xl font-bold text-gray-900, um subtítulo discreto sobre simplicidade e segurança nas transações, e um botão "Começar Agora" com bg-emerald-500 text-white px-8 py-4 rounded-full hover:bg-emerald-600 shadow-lg transform hover:-translate-y-1 transition-all duration-300.

A autenticação será gerenciada pelo Supabase Auth. Teremos rotas separadas para login e registro. A página de login exibirá um formulário centralizado com flex items-center justify-center min-h-screen bg-gray-100. Utilizaremos um Card do Shadcn/UI com w-96 max-w-full. Dentro do card, teremos um Input com label="Email" e placeholder="seu.email@exemplo.com", outro Input para label="Senha" com type="password" e placeholder="********". Abaixo, um Button com variant="default", className="w-full", onClick={handleLogin}, texto "Entrar" e bg-emerald-500 hover:bg-emerald-600. Incluiremos um link para "Esqueci minha senha" e um botão para "Criar Conta" com variant="outline". As respostas do login serão tratadas com toasts: toast.success("Login bem-sucedido!") ou toast.error("Email ou senha inválidos.").

Após o login, o usuário será redirecionado para o dashboard principal, em /dashboard. Este será o hub central do PDV. Imagine um layout com uma sidebar fixa à esquerda (w-64 em bg-gray-800 text-white) e a área de conteúdo principal à direita. A sidebar conterá links de navegação estilizados com px-4 py-3 rounded-lg, hover:bg-gray-700 transition-colors, para: "Vendas" (icon: ShoppingCart), "Estoque" (icon: Package), "Caixa" (icon: dolar-sign), "Pagamentos" (icon: credit-card), "Relatórios" (icon: file-text), e "Configurações" (icon: settings). A área principal do dashboard exibirá um resumo rápido, talvez com cards (Card do Shadcn/UI) usando bg-white shadow-sm rounded-xl p-6 com hover:shadow-md transition-shadow: "Vendas do Dia" (icon: currency text-emerald-500), "Valor em Caixa" (icon: banknote text-blue-500), "Última Sangria" (icon: scissors text-red-500), e talvez um gráfico simples de vendas diárias usando uma biblioteca como Recharts.

A funcionalidade de venda é o core. Na rota /sales, teremos uma interface de frente de caixa. Imagine um layout dividido: à esquerda, uma área para adicionar produtos (busca por código ou nome, Input com debounce e icon: search), e à direita, um resumo do carrinho de compras com quantidade, preço unitário, subtotal, e um botão para remover item. Ao adicionar um item, um toast de sucesso "Produto adicionado ao carrinho!" aparecerá. Abaixo do carrinho, o total da venda, opções de pagamento (botões para "Dinheiro", "Cartão Cielo", "Cartão MercadoPago", "Cartão SumUp"), e um botão "Finalizar Venda" com bg-emerald-500 text-white font-bold py-3 px-6 rounded-lg, hover:bg-emerald-600. Ao clicar em finalizar, um Dialog do Shadcn/UI se abrirá para confirmar a venda e processar o pagamento. Cada opção de pagamento acionará uma função específica. Para pagamento em dinheiro, apenas a confirmação. Para cartões, integrará com as APIs das adquirentes (que podem ser simuladas ou chamadas via Lovable Edge Functions).

Para o painel de controle de estoque, em /inventory, apresentaremos uma tabela (Table do Shadcn/UI) com colunas: "ID Produto", "Nome", "Unidade", "Quantidade em Estoque", "Preço de Custo", "Preço de Venda", "Ações". As ações podem incluir "Editar" (abre um Dialog com Inputs preenchidos) e "Remover" (com confirmação via AlertDialog do Shadcn/UI). Um botão "Novo Produto" (bg-emerald-500 text-white px-4 py-2 rounded-lg) abrirá um modal para cadastro. As operações de estoque terão toasts de feedback como "Produto atualizado com sucesso!" ou "Produto removido.".

A gestão de caixa, em /cashier, será focada nas operações de abertura, fechamento, e sangrias. Ao acessar, se o caixa estiver fechado, um Dialog de abertura se apresentará com um Input para "Saldo Inicial" e um Button "Abrir Caixa". Se aberto, mostrará o "Saldo Atual", "Total de Vendas em Dinheiro", "Total de Sangrias", e botões para "Registrar Sangria" (abre outro Dialog com Input para valor e motivo) e "Fechar Caixa" (abre um Dialog com resumo e confirmação).

Para integrações de pagamento, teremos uma seção /payments. Aqui, o usuário poderá configurar as credenciais para Cielo, MercadoPago e SumUp. Isso pode envolver um formulário com Inputs para API Key, Merchant ID, Token, etc. Estas configurações serão salvas no Supabase. As chamadas para as APIs de pagamento serão feitas através de Lovable Edge Functions para manter a segurança das credenciais e a lógica do backend isolada.

A estrutura das tabelas Supabase inicialmente seria:

users: Gerenciada pelo Supabase Auth.
products: id (uuid, PK), user_id (uuid, FK para auth.users, RLS aplicado), name (text), sku (text, UNIQUE), unit (text), cost_price (numeric), sale_price (numeric), stock_quantity (integer, DEFAULT 0), created_at (timestamptz), updated_at (timestamptz).
sales: id (uuid, PK), user_id (uuid, FK para auth.users, RLS aplicado), total_amount (numeric), payment_method (text: 'cash', 'cielo', 'mercado_pago', 'sumup'), status (text: 'completed', 'pending', 'canceled'), created_at (timestamptz).
sale_items: id (uuid, PK), sale_id (uuid, FK para sales), product_id (uuid, FK para products), quantity (integer), unit_price (numeric), subtotal (numeric).
cash_operations: id (uuid, PK), user_id (uuid, FK para auth.users, RLS aplicado), type (text: 'open', 'close', 'transfer' - sangria/depósito), amount (numeric), initial_balance (numeric, para abertura), final_balance (numeric, para fechamento), reason (text, para sangrias/depósitos), created_at (timestamptz).
payment_configs: id (uuid, PK), user_id (uuid, FK para auth.users, RLS aplicado), provider (text: 'cielo', 'mercado_pago', 'sumup'), credentials (jsonb - para armazenar chaves, IDs, etc.).
Para RLS, por exemplo, na tabela products: CREATE POLICY "Users can only view their own products" ON products FOR SELECT USING (auth.uid() = user_id); CREATE POLICY "Users can insert their own products" ON products FOR INSERT WITH CHECK (auth.uid() = user_id); e similarmente para UPDATE e DELETE.

Implementaremos estados de loading utilizando Skeleton do Shadcn/UI sempre que dados estiverem sendo buscados. Para listas vazias, um Empty State customizado com ilustração e texto como "Nenhum produto cadastrado. Clique em 'Novo Produto' para adicionar um." será exibido. Erros nas chamadas de API serão tratados com toasts genéricos do tipo toast.error("Ocorreu um erro. Por favor, tente novamente.") ou específicos se o erro permitir.

A lógica de abertura/fechamento de caixa no backend (geralmente em Edge Functions ou diretamente no Supabase via triggers/stored procedures) seria responsável por registrar as operações e calcular os saldos. A integração com as adquirentes de cartão envolverá chamadas assíncronas para suas APIs, com feedback em tempo real via polling ou WebSockets (Supabase Realtime se necessário) para status da transação.

Para tornar o design mais memorável e evitar clichés, fugiremos do layout split hero padrão. Na página principal, poderíamos ter um "Massive Typographic Hero" onde o título "Seu Ponto de Venda Inteligente" aparece em text-7xl md:text-9xl font-black text-gray-900 leading-tight, com um subtítulo mais conciso e um CTA destacado. A navegação pode ser mais minimalista, com ícones claros e textos que aparecem em hover. As animações serão sutis, como scroll-triggered reveals para os elementos do dashboard e spring physics para modais e diálogos de confirmação, utilizando framer-motion se desejado. O esquema de cores primário seria um azul profundo (#1E3A8A) para um feeling de profissionalismo e confiança, com emerald-500 como cor de destaque para ações importantes.

A URL pdvlocalpro.lovable.app será o domínio de publicação.
